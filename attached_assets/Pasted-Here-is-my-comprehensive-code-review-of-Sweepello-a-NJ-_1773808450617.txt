Here is my comprehensive code review of Sweepello — a NJ Shore cleaning services brokerage platform built with Express + Drizzle ORM (backend) and React + TanStack Query (frontend).
Architecture Overview
The app is a full-stack monorepo: Express API server, React SPA client, PostgreSQL via Drizzle ORM, Stripe Connect for payments, SendGrid for email, and Replit auth. It has admin, client, and contractor portals with job matching, onboarding, disputes, and review moderation.
CRITICAL Security Issues
1. Unauthenticated admin routes (HIGH severity)
Several routes that manage core business data have no authentication or authorization:
GET /api/clients (line 228) — anyone can list all clients
POST /api/clients (line 233) — anyone can create clients
GET /api/cleaners (line 243) — anyone can list all cleaners
POST /api/cleaners (line 248) — anyone can create cleaners
GET /api/jobs (line 258) — anyone can list all jobs with pricing
POST /api/jobs (line 263) — anyone can create jobs
PATCH /api/jobs/:id (line 293) — anyone can update job status
GET /api/payments (line 320) — anyone can see all payments
POST /api/reviews (line 330) — anyone can post reviews
GET /api/reviews (line 325) — anyone can read all reviews
GET /api/dashboard/stats (line 646) — exposes revenue/profit numbers publicly
GET /api/calendar (line 677) — exposes all job schedules and addresses publicly
GET /api/cleaner-availability/:cleanerId (line 459) — public
GET /api/pricing-estimate (line 104) — exposes internal pricing algorithm publicly
This is the most critical issue. Anyone with the API URL can dump your entire client list, cleaner roster, job history, revenue figures, and payment data.
2. XSS in email templates (MEDIUM severity)
server/sendgrid.ts:44 and :92 — The name and note parameters are interpolated directly into HTML:
<h2>Congratulations, ${name}!</h2>
// and
<p style="...">${note}</p>

If an admin enters <script>alert(1)</script> as a note, or a contractor registers with a malicious name, it gets sent as raw HTML in email.
3. Unvalidated req.body passed directly to storage updates
server/routes.ts:428 — storage.updateServiceRequest(req.params.id, req.body) passes the raw body directly, allowing an attacker to overwrite any column.
server/routes.ts:295 — storage.updateJob(req.params.id, req.body) — same issue.
server/routes.ts:721 — storage.updateContractorOnboarding(userId, req.body) — the contractor can set their own stripeOnboardingComplete: true or onboardingStatus: "complete" by passing it in the body.
4. Empty catch blocks silently swallow errors
server/routes.ts:955 — catch {} after Stripe API call
server/routes.ts:1083 — catch {} after sending approval email
server/routes.ts:1104 — catch {} after sending rejection email
These hide failures. At minimum, log the error.
Design & Architecture Issues
5. Massive monolithic routes.ts (1297 lines)
This single file has ~50 route handlers with duplicated auth checks, duplicated client-creation logic, and duplicated payment-creation logic. Should be split into route modules (e.g., adminRoutes.ts, contractorRoutes.ts, clientRoutes.ts).
6. N+1 query patterns throughout
server/matching.ts:104-113 — Calls storage.getCleanerAvailability(cleaner.id) inside a loop for every cleaner.
server/routes.ts:127-132 — Loops through cleanerIds calling storage.getCleaner(id) individually.
server/routes.ts:450-454 — Loops through offers calling storage.getCleaner(offer.cleanerId) one by one.
server/routes.ts:586-589 — Loops through offers calling storage.getServiceRequest(offer.serviceRequestId) individually.
These will degrade badly as data grows. Use batch queries or JOINs.
7. Duplicated business logic
Client creation logic is copy-pasted in three places:
server/matching.ts:226-241 (in acceptJobOffer)
server/routes.ts:360-375 (in admin service request assignment)
Both create a client with name: "Client" — a placeholder that never gets updated
Payment creation logic is duplicated in:
server/matching.ts:258-273
server/routes.ts:268-284
server/routes.ts:389-404
8. No database foreign keys
The schema in shared/schema.ts defines no foreign key constraints. cleanerId, clientId, jobId, etc. are just plain varchar columns. This means:
No referential integrity — you can delete a cleaner and leave orphaned jobs
No cascade behavior
No database-level protection against invalid references
9. No indexes beyond primary keys
No indexes on frequently queried columns like userId, cleanerId, clientId, status, serviceRequestId. This will cause full table scans as data grows.
10. Stringly-typed enums
Status fields ("pending", "active", "completed", etc.) and role fields ("admin", "client", "contractor") are plain text columns with no enum constraint. Typos like "compled" would silently succeed.
Code Quality Issues
11. Pervasive any types
server/routes.ts:23 — getUserId(req: any) — should use Express Request type
server/routes.ts:27 — isAdmin(req: any)
server/routes.ts:34 — isContractor(req: any)
server/stripeClient.ts:3 — let connectionSettings: any
server/stripeClient.ts:63 — let stripeSync: any = null
server/routes.ts:941 — balance: null as any
server/seed.ts:254 — const paymentRows: any[]
12. Inconsistent authorization patterns
Admin checks are done 3 different ways:
Using the isAdmin() helper function (most routes)
Inline getUserId + getUserProfile + role check (admin routes after line 1060)
Missing entirely (the unauthenticated routes above)
Should use a middleware pattern like requireAdmin that can be applied declaratively.
13. registerRoutes receives but ignores httpServer
server/routes.ts:41-44 — The function takes httpServer as a parameter and returns it, but never uses it inside the function. The routes are registered on app, not httpServer.
14. Unused import
server/routes.ts:2 — createServer is imported but never used.
15. onTimePercent type mismatch in seed vs schema
shared/schema.ts:48 — onTimePercent is integer(), but server/routes.ts:908 passes "100" (a string) when creating a cleaner during onboarding completion. The seed file correctly uses 100 (number).
16. Financial calculations with floating point
Monetary values are stored as decimal in the DB but converted to JavaScript Number for arithmetic (e.g., Number(j.price), Number(cleaner.totalRevenue || 0)). This can cause floating-point precision errors with real money. Use a decimal library or keep values as integers (cents).
17. stripeSync is a module-level singleton with no cleanup
server/stripeClient.ts:63-78 — The lazy singleton stripeSync is never cleaned up, and if getStripeSecretKey() throws after StripeSync is partially initialized, the module is left in a broken state with no way to retry.
18. Inconsistent error responses
Some routes return { message: "..." }, others return { error: "..." } (the webhook handler at server/index.ts:76). The client has to handle both shapes.
Missing Features / Best Practices
19. No tests whatsoever
There are zero test files in the project. For a platform handling financial transactions, contractor agreements, and PII, this is a significant risk.
20. No rate limiting
The public-facing endpoints (contractor application, pricing estimate) have no rate limiting. The application submission endpoint could be abused for spam.
21. No input sanitization on text fields
Free-text fields like notes, specialInstructions, adminNote, comment are stored and presumably rendered without sanitization.
22. No pagination
All list endpoints (getClients(), getJobs(), getPayments(), etc.) return the entire table with no pagination or limits. This will become a performance problem quickly.
23. No logging/audit trail for admin actions
Admin actions (approving contractors, modifying reviews, resolving disputes) have no audit log. You can't trace who did what.
24. Seed data runs on every startup
server/index.ts:128 — seedDatabase() is called on every server start. It does check for existing data, but this pattern is fragile and shouldn't be in the production startup path.
Summary of Priorities
Priority	Issue	Impact
P0	Unauthenticated routes exposing all data	Data breach risk
P0	Raw req.body passed to DB updates	Privilege escalation
P1	XSS in email templates	Email injection
P1	No foreign keys or indexes	Data integrity + performance
P1	No tests	Regression risk
P2	N+1 queries	Performance degradation
P2	Duplicated business logic	Maintenance burden
P2	No pagination	Scalability
P2	Floating-point money math	Financial errors
P3	Code organization (monolithic routes)	Developer velocity
P3	Type safety (any types)	Bug risk
P3	Rate limiting, audit logging	Operational maturity
The codebase shows rapid development velocity — lots of features shipped — but security and engineering fundamentals were deprioritized. The unauthenticated routes are the most urgent fix; everything else in the business logic won't matter if anyone can read your entire database over HTTP.
